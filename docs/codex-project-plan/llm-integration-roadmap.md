# LLM 接入与未完成功能规划

## 1. 当前项目状态

当前项目已经完成反诈助手 MVP 的基础框架，图片上传服务已经独立为单独项目。后续主线工作是把反诈主服务从“占位上传 + 关键词规则分析”升级为“真实文件上传 + OCR/ASR + LLM 风险分析 + 异步任务 + 点数结算”的完整链路。

现有服务结构：

- 反诈主服务：`www/service.antifraud.local.com`
- 独立上传服务：`www/service.storage.company.com`
- 用户端 H5/uni-app：`apps/client`
- 微信小程序端：`apps/mp-wechat`

## 3. 当前未实现功能

### 3.1 反诈服务与独立上传服务打通

当前反诈主服务 `files/upload-token` 仍返回本地占位地址：

```text
/api/v1/files/local-upload-placeholder
```

需要改造为：

- 调用独立上传服务创建上传入口，或直接使用独立上传服务上传接口；
- 将真实文件 `storage_key`、`file_url`、`mime_type`、`file_size` 写入 `file_assets`；
- 保证图片、音频文件都能被后续 OCR/ASR/LLM 使用。

### 3.2 图片 OCR / 视觉理解

当前图片分析依赖前端传入 `text`，或读取 `file_assets.ocr_text`，但没有真实 OCR 流程。

需要实现：

- 根据图片 `file_url` 获取图片内容；
- 调用 OCR 或视觉模型；
- 将识别文字写入 `file_assets.ocr_text`；
- 图片分析无需用户手动补充识别文本。

### 3.3 音频 ASR 转写

当前录音分析依赖前端传入 `text`，或读取 `file_assets.transcript_text`，但没有真实 ASR 流程。

需要实现：

- 根据音频 `file_url` 获取音频内容；
- 调用 ASR 服务；
- 将转写文本写入 `file_assets.transcript_text`；
- 支持较长音频的分段处理。

### 3.4 LLM 风险分析适配器

当前风险报告由关键词规则生成，尚未接入真实 LLM。

需要实现：

- 新增 LLM Client；
- 支持 `LLM_BASE_URL`、`LLM_API_KEY`、`LLM_MODEL` 配置；
- 构造反诈风险分析 prompt；
- 要求模型输出结构化 JSON；
- 校验 JSON 字段；
- 失败时重试；
- 日志脱敏；
- 可降级到本地关键词规则。

### 3.5 分析任务异步化

当前创建分析接口同步生成报告并扣点，后续接入 OCR/ASR/LLM 后容易请求超时。

需要改造为：

- 创建分析记录时状态为 `pending`；
- 队列任务执行 OCR/ASR/LLM；
- 分析中状态为 `processing`；
- 成功后状态为 `completed`；
- 失败后状态为 `failed`；
- 前端通过报告详情接口轮询任务状态。

### 3.6 点数冻结、扣费、退款机制

当前接口创建分析时直接扣点，表中虽然有 `frozen_points`，但尚未真正使用。

需要实现：

- 创建任务时冻结点数；
- 分析成功后确认扣费；
- 分析失败后释放或退回冻结点；
- 点数流水记录冻结、扣费、退款状态；
- 避免 OCR/ASR/LLM 失败时实际扣点。

### 3.7 管理端能力补齐

当前已有管理 API 雏形，但正式上线前还需要：

- 管理端登录；
- 角色和权限；
- 操作日志；
- 查看分析记录；
- 查看 LLM 原始输出和失败原因；
- 手动重试失败任务；
- 风险规则维护。

### 3.8 微信支付真实接入

当前微信支付接口为 mock。

后续需要实现：

- 微信支付下单；
- 支付回调；
- 回调幂等；
- 点数充值流水；
- 失败重放和对账。

### 3.9 日志、监控、限流

后续需要补齐：

- OCR 调用耗时和失败日志；
- ASR 调用耗时和失败日志；
- LLM 调用耗时、错误和 JSON 解析失败日志；
- 分析任务状态变化日志；
- 点数冻结、扣费、退款日志；
- 接口限流；
- 管理端操作日志；
- 生产环境告警。

## 4. 执行计划

### 阶段 1：打通独立上传服务

目标：反诈主服务不再使用本地占位上传地址。

任务：

- 梳理独立上传服务接口返回结构；
- 改造反诈服务文件创建逻辑；
- 保存真实 `storage_key` 和 `file_url`；
- 前端图片页、录音页改为真实上传；
- 报告详情返回真实文件信息。

验收标准：

- 图片能真实上传到独立上传服务；
- 音频能真实上传到独立上传服务；
- `file_assets.file_url` 可访问；
- 图片/音频分析接口能基于真实文件 ID 创建记录。

### 阶段 2：接入图片 OCR / 视觉理解

目标：图片分析不再依赖用户手动填写识别文字。

任务：

- 新增 OCR 或视觉模型配置；
- 新增 OCR Client；
- 图片分析时读取文件 URL；
- 调用 OCR 获取文字；
- 写入 `file_assets.ocr_text`；
- 将 OCR 结果传入风险分析。

验收标准：

- 用户只上传图片，不填写文本，也能生成报告；
- 报告风险点能引用 OCR 识别到的证据文本；
- OCR 失败时记录失败原因，不实际扣点。

### 阶段 3：接入 LLM 风险分析

目标：将关键词 MVP 升级为真实大模型风险分析。

任务：

- 新增 LLM Client；
- 新增 LLM 风险分析 Business；
- 设计并固定 prompt；
- 要求 LLM 输出 JSON；
- 校验输出结构；
- 保存风险等级、风险分、摘要、建议和风险点；
- 保留关键词规则作为辅助输入或失败降级。

验收标准：

- LLM 能稳定返回结构化报告；
- JSON 非法时自动重试一次；
- 重试失败后有明确降级或失败策略；
- 报告文案使用审慎表达，不出现“百分百识别诈骗”等绝对化结论。

### 阶段 4：分析任务异步化

目标：避免分析接口长时间阻塞，支持后台处理和失败重试。

任务：

- 新增分析任务 Job；
- 创建分析记录时写入 `pending`；
- 队列执行时切换为 `processing`；
- 成功后切换为 `completed`；
- 失败后切换为 `failed`；
- 前端创建任务后进入分析中状态；
- 前端轮询报告详情；
- 管理端支持失败重试。

验收标准：

- 创建分析接口快速返回；
- 长耗时 LLM 调用不会阻塞用户请求；
- 失败任务可重试；
- 状态流转符合 `pending -> processing -> completed/failed`。

### 阶段 5：点数冻结、扣费、退款改造

目标：分析成功才实际扣点，失败不扣点。

任务：

- 创建分析任务时冻结点数；
- 点数余额校验包含冻结逻辑；
- 分析成功后确认扣费；
- 分析失败后释放或退款；
- 点数流水记录完整状态；
- 管理端可查看点数变化原因。

验收标准：

- 上传失败不扣点；
- OCR 失败不扣点；
- ASR 失败不扣点；
- LLM 失败不扣点或按降级策略处理；
- 分析成功后点数准确扣除。

### 阶段 6：接入录音 ASR + LLM 分析

目标：录音链路与图片链路一样可真实分析。

任务：

- 新增 ASR 配置；
- 新增 ASR Client；
- 上传音频后调用 ASR；
- 写入 `file_assets.transcript_text`；
- 长音频分段处理；
- 复用 LLM 风险分析逻辑。

验收标准：

- 用户只上传录音，不填写文本，也能生成报告；
- 长音频可以稳定转写；
- 报告风险点能引用转写证据；
- ASR 失败不实际扣点。

### 阶段 7：上线前补强

目标：具备生产灰度能力。

任务：

- 管理端鉴权；
- 管理端角色权限；
- 管理端操作日志；
- 微信支付真实接入；
- API 限流；
- LLM/OCR/ASR 调用日志；
- 生产告警；
- 隐私协议和免责声明检查；
- 删除记录后的文件处理策略；
- 数据库备份；
- 对象存储生命周期策略。

验收标准：

- 生产环境关闭 `APP_DEBUG`；
- 关键接口具有限流；
- 管理端不可裸访问；
- 支付回调幂等；
- 分析失败可追踪；
- 用户隐私和敏感信息有脱敏策略。

## 5. 推荐下一步

建议下一步先执行阶段 1：打通独立上传服务。

原因：

- 文件 URL 是 OCR、ASR、LLM 的前置输入；
- 当前反诈主服务仍使用上传占位地址；
- 只有文件链路稳定，后续 AI 分析才能逐步接入；
- 这一阶段风险可控，适合作为后续所有 LLM 工作的第一步。

阶段 1 完成后，再依次推进图片 OCR、LLM 风险分析、异步队列和点数冻结扣费。
